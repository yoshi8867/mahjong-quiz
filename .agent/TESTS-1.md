## 사전 메모 (실행 모드용)
- 파서 API 이름은 구현에서 정해진다. 실행 시 우선 `browser_evaluate`로
  `Object.keys(window).filter(k => /tile|mj|mahjong|parse|hand/i.test(k))`
  를 평가해 노출된 전역(파서/네임스페이스)을 찾는다. 아래 케이스는 대표 호출을
  `parseHand(...)`로 적지만, 실제 노출명(예: `window.MJ.parse`, `window.Tiles.parse`)으로 치환해 평가한다.
- 파서가 전역 함수로 노출되지 않으면(모듈 스코프) 해당 파싱 케이스는 쇼케이스에
  렌더된 DOM 결과(타일 개수/속성)로 대신 판정한다. 판정 근거를 TESTRUN에 명시.
- 표기 규약(판정 기준): 만수 `m`, 통수 `p`, 삭수 `s`, 자패 `z`(1z=동,2z=남,3z=서,4z=북,5z=백,6z=발,7z=중), 적도라 `0m/0p/0s`(=빨간 5).
- 유니코드 마작패 코드포인트: 만수 1~9 = U+1F007~U+1F00F, 삭수 1~9 = U+1F010~U+1F018, 통수 1~9 = U+1F019~U+1F021, 자패는 U+1F000(동)~U+1F006 순서가 **동남서북·중·발·백**이므로 z표기(백=5z가 U+1F006, 발=6z가 U+1F005, 중=7z가 U+1F004)와 순서가 어긋난다 → 매핑 정확성 확인 대상.

### T1.1: 표준 표기 정상 파싱
- 절차: 콘솔에서 `parseHand("123m456p789s11z")` 평가. 결과 배열 길이와 각 원소의 suit/num 필드를 확인한다.
- 기대: 원소 11개. 순서대로 (m1,m2,m3,p4,p5,p6,s7,s8,s9,z1,z1). 각 원소가 수트(m/p/s/z)와 숫자(1~9)를 구분해 담고 있고, 파싱 순서가 입력 순서와 일치한다.

### T1.2: 적도라(0m/0p/0s) 파싱 — 빨간 5로 인식
- 절차: `parseHand("0m0p0s")` 평가. 각 원소의 숫자값과 적도라 표식(예: `red:true` 또는 `aka` 플래그)을 확인한다.
- 기대: 원소 3개, 각각 숫자값은 5로 정규화되고(랭크상 5로 취급) 적도라 플래그가 true. 일반 `5m5p5s`를 파싱한 결과와 숫자는 같지만 적도라 플래그로 구분된다.

### T1.3: 자패 전체(1z~7z) 매핑 정확성
- 절차: `parseHand("1234567z")` 평가 후, 쇼케이스에서 각 자패가 렌더된 유니코드 글리프의 코드포인트를 확인한다. 특정 글리프 요소의 텍스트를 `[...el.textContent][0].codePointAt(0).toString(16)`로 평가.
- 기대: 파싱 결과 자패 7개(z1~z7). 렌더 매핑이 1z=동(U+1F000), 2z=남(U+1F001), 3z=서(U+1F002), 4z=북(U+1F003), 5z=백(U+1F006), 6z=발(U+1F005), 7z=중(U+1F004)로 정확. (백/발/중 순서 역전 실수 없음)

### T1.4: 파서 엣지케이스 — 잘못된/빈 표기
- 절차: 다음을 각각 평가하고 반환값 또는 throw 여부를 기록한다: (a) `parseHand("")` (b) `parseHand("8z")` (c) `parseHand("9z")`/`parseHand("0z")` (d) `parseHand("123")` (수트 접미어 없음) (e) `parseHand("m123")` (숫자 앞 수트) (f) `parseHand("12x3m")` (미지 수트 x) (g) `parseHand("1.5m")`/`parseHand("!!")`.
- 기대: (a) 빈 배열 반환(또는 명시적 빈 결과) — 예외로 페이지가 죽지 않음. (b),(c) 자패 범위 밖(8z/9z/0z)은 거부: 예외 throw 또는 무시/빈결과 등 **일관된 방어 동작**이며 잘못된 타일을 만들지 않는다. (d),(e),(f),(g) 유효하지 않은 입력은 타일을 생성하지 않거나 예외를 던지되, 콘솔 미처리 예외로 페이지 전체가 중단되지 않는다. 각 입력의 실제 반환/예외를 TESTRUN에 표로 기록.

### T1.5: 34종 타일 2방식(유니코드 vs CSS) 나란히 비교 [수용1]
- 절차: 쇼케이스 로드 후 snapshot으로 "유니코드"/"CSS 커스텀"(또는 동등) 라벨의 비교 섹션을 찾는다. 각 방식 줄의 타일 요소 수를 `document.querySelectorAll` 계열로 세어 확인.
- 기대: 34종(만1-9, 통1-9, 삭1-9 = 27 + 자패 7 = 34) 타일이 유니코드 방식 한 줄, CSS 커스텀 방식 한 줄로 각각 34개씩 표시되어 나란히 비교 가능. 두 줄의 타일이 같은 종류·같은 순서로 대응된다.

### T1.6: CSS 커스텀 타일 몸체 렌더 [수용1]
- 절차: CSS 커스텀 타일 요소 하나를 골라 `getComputedStyle`로 배경색/모서리 반경/그림자/크기(`--tile-w`,`--tile-h` 반영)를 확인. 내부에 글리프 또는 한자/숫자 텍스트가 있는지 snapshot으로 본다.
- 기대: 흰/아이보리 몸체(투명 배경 아님), border-radius > 0, box-shadow 존재, 폭·높이 > 0. 내부에 해당 타일을 나타내는 텍스트(유니코드 글리프 또는 한자/숫자)가 있어 어떤 패인지 식별 가능. 치수·색이 하드코딩이 아닌 CSS 변수 토큰에서 유래(폭이 `--tile-w`와 일치).

### T1.7: 적도라 시각 구분 [수용4]
- 절차: 쇼케이스에서 일반 5(예: 5m)와 적도라 0m이 함께 보이는 구역을 snapshot으로 확인. 두 타일 요소의 글자색/배경을 `getComputedStyle`로 비교(적색 계열 `--color-accent` 반영 여부).
- 기대: 적도라(0m/0p/0s)의 숫자/글리프가 빨강 계열로 렌더되어 일반 5(m/p/s)와 색으로 명확히 구분된다. 두 방식(유니코드·CSS 커스텀) 모두에서 구분 가능하거나, 최소한 CSS 커스텀 방식에서 구분된다.

### T1.8: 90도 회전 타일 — 퐁/치 세트 [수용2]
- 절차: "퐁/치"(또는 부른 패) 세트 샘플을 snapshot으로 찾는다. 회전된 타일 요소의 변환을 `getComputedStyle(el).transform`으로 확인(회전 행렬). 회전 타일과 비회전 타일의 렌더 박스(`getBoundingClientRect`) 비율을 비교.
- 기대: 세트 내 정확히 1장이 90도 회전(transform에 rotate 90/270deg 또는 그에 해당하는 matrix). 회전 타일의 시각적 폭·높이가 원본 타일의 높이·폭과 뒤바뀐 형태로 자연스럽고, 다른 타일과 세로 정렬이 어긋나거나 잘리지 않는다.

### T1.9: 버림패 강 — 6장씩 줄바꿈 [수용2]
- 절차: 버림패(강) 샘플 구역을 찾는다. 7장 이상(예: 8~14장)을 담은 강의 타일 요소들의 `getBoundingClientRect().top` 값을 모아, 같은 top 값(=같은 줄)당 타일 수를 계산한다.
- 기대: 한 줄에 최대 6장, 7번째부터 다음 줄로 줄바꿈된다(첫 줄 top 동일한 타일 6개 → 다음 줄 새 top). 줄 간 타일이 겹치지 않고 정렬이 일정.

### T1.10: 손패 13 + 쯔모 1(간격 분리) + 도라 표시패 [수용3]
- 절차: 손패 레이아웃 샘플 구역을 찾는다. 손패 타일 수를 세고, 마지막(쯔모) 타일과 그 앞 타일 사이 좌우 간격을 앞쪽 인접 타일 간격과 `getBoundingClientRect`로 비교. 도라 표시패 구역이 별도로 존재하는지 확인.
- 기대: 손패 13장이 균등 간격으로 가로 나열되고, 14번째(쯔모) 타일은 앞선 간격보다 눈에 띄게 큰 간격으로 분리되어 있다. 별도로 도라 표시패(1장 이상)가 표시된다.

### T1.11: 콘솔 에러 0건
- 절차: 쇼케이스 로드 직후 `browser_console_messages(level:"error")` 확인. 각 파서 엣지 케이스(T1.4) 평가 후에도 재확인.
- 기대: 페이지 로드 및 정상 렌더 과정에서 콘솔 error 0건. (T1.4의 의도된 예외를 콘솔에 던지더라도, 그것이 페이지 로드/렌더를 막지 않아야 하며 로드 시점 에러는 0건.)

### T1.12: 반응형 레이아웃 — 375px / 1280px 겹침·잘림 없음
- 절차: `browser_resize`로 375×812(모바일), 1280×800(데스크톱) 두 뷰포트에서 각각 snapshot + 스크린샷. 강/손패/세트 구역에서 타일 요소가 뷰포트 밖으로 잘리거나(우측 clientWidth 초과) 서로 겹치는지 확인(`getBoundingClientRect` 교차 검사 또는 육안).
- 기대: 두 뷰포트 모두에서 타일이 가로 스크롤 없이(또는 의도된 스크롤 컨테이너 내에서) 표시되고, 타일 간 겹침·글리프 잘림이 없다. 강 6장 줄바꿈 규칙이 모바일에서도 유지된다.
